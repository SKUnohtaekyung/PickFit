## 0. 문서 목적

이 문서는 **픽핏의 디자인 시스템 단일 기준서**다.
목표는 다음 3가지다.

1. 디자이너, 프론트엔드, Codex가 **같은 시각 언어**를 사용하게 만든다.
2. 기능 추가 시에도 화면이 흔들리지 않도록 **토큰, 규칙, 컴포넌트, 패턴**을 고정한다.
3. 감이나 취향이 아니라 **제품 목표, 접근성, 상호작용 원칙, 구조화된 UX 기준**에 따라 의사결정하게 만든다.

이 문서는 다음 상황에서 기준 문서로 사용한다.
- 화면 설계
- UI 컴포넌트 제작
- 프론트엔드 토큰 정의
- QA / 접근성 점검
- 신규 기능 확장 시 일관성 유지

## 1. 이 문서의 우선순위와 사용 규칙

### 1.1 우선순위
1. 이 문서
2. `project.md`
3. 확정된 화면 시안
4. 일회성 대화나 임시 메모

### 1.2 금지 사항
- 브랜드 컬러를 컴포넌트 상태색처럼 무분별하게 재해석하지 않는다.
- 새로운 라운드, 그림자, 간격, 폰트 크기를 임의로 추가하지 않는다.
- 로고 조형을 UI 아이콘처럼 변형해 사용하지 않는다.
- AI 기능을 이유로 복잡한 미래형 인터랙션을 억지로 추가하지 않는다.
- “더 화려해 보이게”를 이유로 정보 위계를 무너뜨리지 않는다.

### 1.3 기본 원칙
- 픽핏은 **패션 콘텐츠 앱**이 아니라 **패션 결정 엔진**이다.
- 화면은 “구경”보다 “결정”을 빠르게 만들어야 한다.
- UI는 예쁘기보다 먼저 **빠르고 신뢰할 수 있어야** 한다.
- 설명은 감성 카피보다 먼저 **근거와 제약을 분명히 보여야** 한다.

## 2. 제품 맥락 요약

### 2.1 제품 정의
픽핏은 사용자의 상황, 취향, 체형 고민, 예산을 빠르게 파악하고,
정규화된 상품 및 리뷰 데이터를 바탕으로 **완성 코디와 추천 이유를 설명해 3분 안에 구매 결정을 끝내게 해주는 웹앱**이다.

### 2.2 UI가 반드시 지원해야 하는 핵심 가치
- 빠른 선택
- 설명 가능한 추천
- 핏 실패 감소
- 비교의 자동화
- 구매 직전의 불안 감소

### 2.3 UI 성격
픽핏의 UI는 아래 어느 한쪽으로 치우치면 안 된다.

#### 피해야 할 방향
- 과도하게 여성향 코디앱 같은 말랑한 감성
- 너무 금융앱 같은 차갑고 건조한 구조
- 대형 커머스처럼 정보가 과밀한 리스트 중심 화면
- 미래형 AI 데모처럼 과장된 글로우/네온/복잡한 애니메이션

#### 지향하는 방향
**Urban Smart Fashion Utility**
- 도시적
- 정돈됨
- 빠름
- 신뢰감 있음
- 패션 감도는 있으나 과장되지 않음
- 테크 기반이지만 사람 중심임

## 3. 디자인 원칙

### 3.1 Decision over Discovery
픽핏의 UI는 탐색보다 결정을 돕는 방향으로 설계한다.

의미:
- 상품 나열보다 추천 결과의 구조가 중요하다.
- 리스트 스크롤 양을 줄이고 추천 카드의 설명 밀도를 높인다.
- 사용자가 “다 봤다”가 아니라 “결정했다”를 느끼게 한다.

### 3.2 Outfit over Item
추천 단위는 기본적으로 상품 1개보다 **룩 / 코디**가 우선이다.

의미:
- 결과 화면의 1차 단위는 완성 코디 카드다.
- 상품 카드는 코디 안의 구성 요소 또는 대안으로 내려간다.
- UI 계층도 `Outfit -> Item -> Option` 순서를 따른다.

### 3.3 Explain before Convince
추천 결과는 설득보다 설명이 먼저다.

의미:
- “왜 추천했는지”를 먼저 보여준다.
- 핏/사이즈/소재/배송/반품/리뷰 근거를 구조적으로 배치한다.
- 감정적 카피보다 조건과 근거를 우선한다.

### 3.4 Trust is Visible
신뢰는 숨겨진 로직이 아니라 화면에 드러나는 정보에서 생긴다.

의미:
- 반영한 조건, 제외한 조건, 데이터 기준 시점을 보여준다.
- AI가 아직 모르는 것은 모른다고 명시한다.
- 민감정보 자동 입력 등 하지 못하는 행동은 사전에 고지한다.

### 3.5 Calm Density
정보는 충분해야 하지만 과밀하면 안 된다.

의미:
- 패션 커머스는 이미지와 텍스트가 모두 중요하다.
- 정보량은 확보하되 그룹화와 위계로 부담을 줄인다.
- 카드 내부에서 텍스트는 3단계 위계 이내로 제한한다.

### 3.6 Soft Geometry, Hard Logic
형태는 부드럽되 구조는 단단해야 한다.

의미:
- 둥근 라운드와 부드러운 색조를 사용하되,
- 정렬, 간격, 비율, 반복 규칙은 매우 엄격해야 한다.

## 5. 로고 시스템

### 5.1 최종 로고 방향
현재 확정 로고 방향은 **3순위 안 기반의 ㅍㅍ 모노그램**이다.

컨셉:
- 한글 초성 `ㅍㅍ`를 사용해 `픽핏`의 기억 구조를 심볼화한다.
- 두 구조체는 단순 반복이 아니라 브랜드 리듬을 만든다.
- 하단 라임 포인트는 “선택 지점 / 결정 포인트 / 맞춤의 기준점”을 암시한다.

### 5.2 로고 구성
1. 심볼: `ㅍㅍ` 모노그램
2. 워드마크: `픽핏`
3. 서브카피: `내 상황에 맞는 코디를 고르는 가장 빠른 방법`

### 5.3 로고 사용 원칙
- 로고는 기본적으로 **워드마크 우선**이 아니라 **심볼 + 워드마크 락업**으로 사용한다.
- 앱 아이콘, 파비콘, 프로필 이미지는 **심볼 단독** 사용을 원칙으로 한다.
- 서브카피는 제품 소개용 / 피치덱용 / 랜딩 상단용에만 사용하고, 작은 UI에 상시 노출하지 않는다.

### 5.4 로고 배경
기본 로고 배경은 `#FFFFFF` 또는 `#F7F8FC`를 사용한다.

예외:
- 앱 아이콘
- 스플래시
- 브랜드 히어로 영역
이 경우 Brand Blue 배경 위에 로고 심볼을 얹을 수 있다.

### 5.5 로고 최소 여백
로고 주변 최소 여백은 **심볼 하단 라임 정사각형 높이의 4배**를 기준으로 한다.

실무 규칙:
- 다른 그래픽, 텍스트, 테두리, 버튼이 이 영역 안으로 침범하면 안 된다.

### 5.6 로고 최소 크기
- 풀 락업 로고: 폭 160px 이하로 줄이지 않는다.
- 워드마크만: 폭 96px 이하로 줄이지 않는다.
- 심볼만: 20px 미만 금지, 앱 아이콘용 최소 32px.

### 5.7 허용 버전
- 기본 컬러 버전
- 모노톤 블랙 버전
- 반전 화이트 버전
- 심볼 단독 버전
- 수평형 락업
- 수직형 락업

### 5.8 금지 사항
- 라임 포인트 색상 변경 금지
- 심볼 비율 임의 변경 금지
- 심볼 내부 획 굵기 개별 수정 금지
- 드롭 섀도우, 그라데이션, 글로우 효과 금지
- 외곽선 추가 금지
- 로고 위에 패턴/사진/텍스처 직접 배치 금지
- 자간 임의 수정 금지

## 6. 컬러 시스템

### 6.1 컬러 사용 철학
픽핏의 색은 단순 장식이 아니라 **기능적 위계**를 가진다.

- 블루: 브랜드와 행동
- 라임: 선택 순간의 강조
- 뉴트럴: 정보 구조와 읽기 편함

브랜드 컬러는 모든 컴포넌트 상태를 대체하지 않는다.
상태 표현은 의미와 접근성을 기준으로 운영한다.

---

### 6.2 핵심 브랜드 컬러

#### Brand Blue
- Hex: `#6574FF`
- 역할: 로고, 앱 아이콘, 브랜드 그래픽, 대형 하이라이트 카드, 히어로 섹션
- 인상: 신뢰, 세련, 부드러운 기술감

#### CTA Blue
- Hex: `#4D5EFF`
- 역할: 주요 버튼, 활성 탭, 선택 상태, 진행바, 주요 인터랙션
- 인상: 행동 유도, 명확한 반응

#### Accent Lime
- Hex: `#E4FF5D`
- 역할: 배지, 추천 포인트, 숫자 강조, 선택 기준점, 상태 포인트
- 인상: 선택의 에너지, 기억점, 패션 포인트

---

### 6.3 뉴트럴 컬러

#### Ink
- Hex: `#12141A`
- 역할: 제목, 숫자, 핵심 정보, 강한 본문

#### Text Secondary
- Hex: `#5F6675`
- 역할: 본문, 설명, 보조 라벨, 메타 정보

#### Border
- Hex: `#D9DCE6`
- 역할: 구분선, 인풋 보더, 카드 경계, 비활성 상태 테두리

#### Background
- Hex: `#F7F8FC`
- 역할: 앱 기본 배경, 기본 캔버스

#### Surface Subtle
- Hex: `#EEF1F6`
- 역할: 서브 카드 배경, 선택 전 칩 배경, 약한 분리면

---

### 6.4 기능성 상태 컬러
브랜드 컬러는 상태 색을 완전히 대체하지 않는다.
상태 시스템은 의미와 접근성을 기준으로 별도 운영한다.

#### Success
- Strong: `#1F8A4D`
- Surface: `#EAF7EF`
- Use: 저장 완료, 적용 완료, 추천 조건 충족, 재고 충분

#### Warning
- Strong: `#B7791F`
- Surface: `#FFF4E5`
- Use: 사이즈 주의, 반품 조건 주의, 배송 지연 가능성, 정보 일부 불확실

#### Error
- Strong: `#C53B3B`
- Surface: `#FDECEC`
- Use: 로딩 실패, 링크 오류, 선택 불가, 품절

#### Info
- Strong: `#3157E0`
- Surface: `#EAF0FF`
- Use: 가이드, 설명, 비교 도움말, AI 참고 정보

규칙:
- 성공/경고/오류는 의미 전달용이며 브랜드 정체성 표현용으로 쓰지 않는다.
- 텍스트와 아이콘은 항상 충분한 대비를 확보한다.

---

### 6.5 컬러 사용 비율
브랜드 화면 전체 기준 권장 비율:
- Background / Neutral Surfaces: 60
- Brand / CTA Blue family: 30
- Accent Lime: 10 이하

설명:
- 라임은 메인 면색이 아니라 **포인트 신호**다.
- 라임 사용량이 많아지면 패션 감도보다 캠페인/이벤트 느낌이 강해진다.
- 블루는 신뢰를 만들지만 남용하면 금융/생산성 앱처럼 보일 수 있으므로 뉴트럴 면적을 충분히 확보한다.

---

### 6.6 컬러 조합 규칙

#### 허용 조합
- CTA Blue 버튼 + White 텍스트
- Brand Blue 배경 + White 심볼
- Background + Ink 텍스트
- Surface Subtle + Ink / Text Secondary
- Accent Lime + Ink 텍스트 (작은 배지에서만)
- Warning Surface + Warning Strong
- Error Surface + Error Strong

#### 주의 조합
- Accent Lime 위 White 텍스트: 금지에 가깝다. 대비 부족 가능성이 높다.
- Border 위 Text Secondary만으로 정보 강조: 약하다.
- Brand Blue와 CTA Blue를 한 영역에서 동등 비중으로 병치: 시각 hierarchy 붕괴 위험.

## 7. 타이포그래피 시스템

### 7.1 기본 서체
- Primary Font: **Pretendard Variable**

선정 이유:
- 웹앱에서 가독성이 높다.
- 한글/영문 혼용 안정성이 좋다.
- 숫자, 필터, 가격, 비교표 등 정보 구조에서 성능이 좋다.
- 패션 서비스에 필요한 미니멀한 인상과 테크 제품의 정돈감을 동시에 만족한다.

### 7.2 타이포 사용 원칙
- 타이포는 “브랜드 멋”보다 **정보 읽힘**이 우선이다.
- 헤드라인은 압축감 있게, 본문은 숨 쉬게 한다.
- 숫자, 가격, 조건, 비교 정보는 문장보다 한 단계 더 빠르게 읽히게 한다.

### 7.3 자간과 행간
현재 브랜드 기준:
- 기본 행간: `140%`
- 헤드라인/타이틀 추적값: `-2%`

실무 적용 규칙:
- Display / Hero / 큰 제목: `-2%`
- 일반 제목: `-1% ~ -2%`
- 본문 / 설명 / 메타: `0%` 또는 `-1%`

이유:
- 한글 본문 전체를 `-2%` 고정하면 작은 화면에서 답답해질 수 있다.

### 7.5 숫자와 가격 표현
- 가격, 배송일, 할인율, 리뷰 수치는 본문보다 한 단계 강조한다.
- 긴 가격 표기는 쉼표를 사용한다. 예: `89,000원`
- 할인율은 항상 숫자와 `%`를 붙여 표기한다. 예: `12%`
- 비교표에서 핵심 숫자는 `Ink` 또는 `CTA Blue`로 강조하고, 보조 라벨은 `Text Secondary` 사용.

## 8. 간격 시스템

### 8.1 기초 단위
기본 spacing unit은 **4px**이다.

이유:
- 디지털 UI에서 충분히 세밀하면서도,
- 8px 리듬과 조합이 쉽고,
- 모바일 화면에서 padding 조절이 용이하다.

### 8.2 Spacing token
- `space-1 = 4px`
- `space-2 = 8px`
- `space-3 = 12px`
- `space-4 = 16px`
- `space-5 = 20px`
- `space-6 = 24px`
- `space-8 = 32px`
- `space-10 = 40px`
- `space-12 = 48px`
- `space-16 = 64px`

### 8.3 사용 규칙
- 텍스트와 텍스트 사이 기본: 8px 또는 12px
- 카드 내부 패딩 기본: 16px 또는 20px
- 섹션 간 간격 기본: 24px 또는 32px
- 큰 섹션 분리: 40px 이상
- 모바일 하단 CTA 전 여백: 최소 24px

## 9. 라운드, 테두리, 그림자

### 9.1 형태 원칙
픽핏의 형태 언어는 **Soft Geometry**다.
즉, 지나치게 각지지 않되 유아 앱처럼 둥글지도 않다.

### 9.2 Radius token
- `radius-xs = 8px`
- `radius-sm = 12px`
- `radius-md = 16px`
- `radius-lg = 20px`
- `radius-xl = 24px`
- `radius-pill = 999px`

### 9.3 권장 매핑
- 버튼 / 인풋 / 셀렉터: `12px`
- 카드 기본: `16px`
- 대형 카드 / 하이라이트 카드: `20px`
- 모달 / 바텀시트 상단 모서리: `24px`
- 칩 / 필터 태그: `999px`
- 앱 아이콘: 운영체제 가이드에 맞추되 시안상 22~24px 수준의 둥근 정사각형 감각 유지

### 9.4 테두리
- 기본 보더: `1px solid var(--pf-border)`
- 강조 보더: `1px solid var(--pf-cta-blue)`
- 위험/경고 보더: 의미 색상 Strong 사용

### 9.5 그림자 원칙
픽핏은 신뢰형 제품이므로 그림자를 과장하지 않는다.
기본 우선순위는 **레이어 > 보더 > 배경면 차이 > 아주 약한 그림자** 순서다.

#### Shadow 1
- Use: 떠 있는 카드, 드롭다운, 바텀시트
- Example: `0 4px 16px rgba(18, 20, 26, 0.06)`

#### Shadow 2
- Use: 모달, 중요한 오버레이
- Example: `0 12px 32px rgba(18, 20, 26, 0.10)`

금지:
- 블루/라임 컬러 그림자
- 카드마다 그림자를 다르게 쓰는 것
- 피그마에서 임의의 blur 기반 글로우 남용

## 12. 이미지 / 사진 시스템

### 12.1 제품 이미지 원칙
- 추천 결과의 핵심은 실제 상품 이해다.
- 이미지가 작아져도 실루엣, 핏, 컬러가 읽혀야 한다.

### 12.2 제품 이미지 기준
- 배경이 너무 복잡한 이미지는 우선순위 낮춤
- 정면/착용컷 우선
- 모델컷 + 제품컷 조합이 가능하면 가장 좋음
- 썸네일에서 핏/길이감을 읽기 어려운 컷은 보조 이미지로만 사용

### 12.3 코디 카드 이미지 원칙
- 코디 카드에는 가능하면 조합 결과를 한 번에 상상할 수 있게 대표 이미지를 둔다.
- 가능하면 메인 상품 1개만 크게 보여주지 말고, 전체 룩 방향성을 이해시킬 수 있는 조합 이미지를 제공한다.

## 13. 모션 시스템

### 13.1 모션 철학
픽핏의 모션은 “와 멋지다”보다 **“지금 무엇이 진행 중인지 안다”**를 위해 존재한다.

### 13.2 모션 원칙
- 짧고 명확할 것
- 피로를 유발하지 않을 것
- AI 진행 상태를 설명하는 데 도움 될 것
- 페이지 전환보다 상태 전환에 집중할 것

### 13.3 Duration token
- fast: 120ms
- normal: 180ms
- emphasis: 240ms
- panel / sheet: 280ms

### 13.4 Easing
- 기본: `cubic-bezier(0.2, 0.8, 0.2, 1)`
- 감속 진입: `ease-out`
- 상태 강조: overshoot 없이 안정적 curve 사용

### 13.5 모션 사용처
- 버튼 press feedback
- 칩 선택 상태 전환
- 탭 전환
- AI 진행 상태 step 변화
- 바텀시트 / 모달 등장
- 스켈레톤 -> 콘텐츠 치환

### 13.6 금지 모션
- 패럴랙스
- 과한 bounce
- 루핑되는 decorative animation
- 로딩 중 의미 없는 반짝임

## 14. 접근성 기준

### 14.1 기본 기준
픽핏은 최소 **WCAG 2.2 AA** 수준을 기준으로 설계한다.

핵심 적용:
- 일반 텍스트 대비 최소 4.5:1
- 큰 텍스트는 3:1 가능
- 포인터 타깃은 최소 24x24 CSS px가 필요하며,
- 실제 제품 기준은 Apple 44x44pt, Material 48x48dp를 참고해 **48x48px hit area**를 기본 목표로 한다.

### 14.2 인터랙티브 요소 최소 목표
- 텍스트 버튼 최소 높이: 40px
- 주요 버튼 최소 높이: 48px
- 아이콘 버튼 클릭 영역: 48x48px
- 칩, 탭, 토글, 드롭다운 항목도 48px 기준을 우선 적용

### 14.3 포커스 상태
- 키보드 탐색이 가능한 화면에서는 focus ring을 반드시 제공한다.
- Focus ring 컬러: `var(--pf-cta-blue)`
- Focus ring thickness: 2px
- Offset: 2px

### 14.4 의미 전달 방식
- 색만으로 상태를 전달하지 않는다.
- 경고/오류/성공은 아이콘 + 레이블 + 보조 문장을 함께 사용한다.

### 14.5 모션 접근성
- OS의 reduced motion 설정을 존중한다.
- reduced motion에서는 skeleton shimmer와 panel slide를 축소 또는 제거한다.

### 14.6 텍스트 접근성
- Body 14px 미만 본문 금지
- 긴 문단은 2~4문장 단위로 끊어 사용
- 숫자/조건/가격 정보는 리스트나 표 구조로 분리

## 15. 컴포넌트 시스템

이 섹션은 Codex와 프론트엔드가 실제로 구현해야 하는 핵심 컴포넌트를 정의한다.

---

### 15.1 Button

#### 목적
행동 유도, 다음 단계 이동, 링크 이동, 저장, 비교 진입 등

#### Button hierarchy
1. Primary
2. Secondary
3. Tertiary / Ghost
4. Destructive
5. Icon Button

#### Primary Button
- Background: `CTA Blue`
- Text: White
- Radius: 12px
- Height: 48px default / 56px large
- Padding: 0 16px or 0 20px
- Use: 다음, 추천 보기, 비교 시작, 구매 링크 이동

#### Secondary Button
- Background: White or `Background`
- Border: `Border`
- Text: `Ink`
- Use: 필터 열기, 대안 보기, 저장 전 비교

#### Tertiary / Ghost
- Background: transparent
- Text: `CTA Blue` or `Ink`
- Use: 보조 액션, 링크성 액션, 상세 더보기

#### Destructive
- Background: `Error`
- Text: White
- Use: 삭제, 초기화, 저장 해제

#### Icon Button
- Size: 48x48px
- Radius: 12px or pill depending on context
- Use: 저장, 공유, 닫기, 뒤로가기, 필터 토글

#### States
- Default
- Hover
- Pressed
- Focused
- Disabled
- Loading

#### Disabled rule
- Primary disabled background: `Border`
- Text: White at reduced opacity or muted gray on light button
- Disabled는 항상 시각적으로 눌리지 않아야 한다.

#### Loading rule
- 스피너는 텍스트를 완전히 밀어내지 않고, 가능하면 왼쪽에 함께 두거나 중앙 정렬 구조로 유지한다.
- 로딩 중 버튼 폭은 변하지 않게 한다.

---

### 15.2 Text Field / Input

#### 목적
텍스트 질문, 검색, 조건 세부 입력, 사이즈/예산 상세 입력

#### 구조
- Label
- Field container
- Optional prefix/suffix
- Assistive text
- Error text

#### 스타일
- Height: 48px minimum
- Radius: 12px
- Border: `Border`
- Background: White
- Text: `Ink`
- Placeholder: `Text Secondary`

#### 상태
- Default
- Focus
- Filled
- Error
- Disabled

#### 규칙
- 검색/대화 입력창은 기본 인풋보다 크게 보여도 좋지만, placeholder를 과도한 문장형으로 넣지 않는다.
- 예산 등 수치 입력은 자유 입력보다 **선택지 + 필요한 경우 세부 입력** 구조를 우선한다.

---

### 15.3 Choice Chip / Filter Chip

#### 목적
상황, 무드, 색, 핏, 예산대, 피하고 싶은 요소, 피드백 선택

#### 스타일
- Height: 36px ~ 40px
- Padding: 0 12px
- Radius: pill
- Default background: `Surface Subtle`
- Default text: `Ink`
- Selected background: `EAF0FF` or `CTA Blue`
- Selected text: `Ink` on light selected or White on strong selected

#### 규칙
- 온보딩에서는 toggle chip보다 **single/multi select chip**이 핵심이다.
- 슬라이더보다 범주형 chip을 우선 사용한다.
- 너무 많은 선택지가 있으면 카테고리 그룹으로 나눈다.

---

### 15.4 Segmented Control

#### 목적
결과 보기 모드 전환
예: 코디 / 상품 / 비교

#### 규칙
- 2~4개 옵션까지만 사용
- 각 세그먼트 최소 hit area 48px
- 선택 상태는 fill + text weight 변화로 명확히 보여준다

---

### 15.5 Tab / Navigation

#### Top tab
- Use: 결과 화면의 상위 정보 섹션 전환
- 예: 추천 / 비교 / 저장

#### Bottom navigation
MVP에서는 과도한 5탭 구조보다 단순 구조를 권장한다.

권장 탭 예시:
- 홈
- 추천 결과 / 저장
- 내 정보

규칙:
- 네비게이션 항목은 3개 중심으로 유지
- 정보 구조를 UI 깊이보다 우선한다.

### 15.6 Card System

#### 원칙
픽핏에서 카드 하나는 단순 박스가 아니라 **결정 단위**다.
따라서 카드 내부에는 항상 “무엇인지 / 왜 중요한지 / 지금 무엇을 할 수 있는지”가 들어가야 한다.

#### Card hierarchy
1. Outfit Recommendation Card
2. Product Item Card
3. Comparison Card
4. Evidence / Reason Card
5. Feedback Card
6. Saved Card

---

#### Outfit Recommendation Card
가장 중요한 카드.

필수 정보:
- 코디 이름 또는 상황 라벨
- 대표 이미지
- 코디 구성 요약
- 추천 이유 2~3줄
- 가격 범위 또는 대표 총액
- 핏/리스크 배지
- CTA: 자세히 보기 / 비교하기 / 구매 링크

권장 구조:
1. Header: 상황/무드/적합도
2. Visual: 이미지
3. Body: 왜 추천했는지
4. Meta: 가격, 배송, 반품, 요약 리뷰
5. Actions

#### Product Item Card
필수 정보:
- 이미지
- 브랜드
- 상품명
- 가격 / 할인
- 핏 / 소재 / 계절감 태그
- 리뷰 요약 1줄
- 저장 / 비교 / 링크 이동

#### Comparison Card
필수 정보:
- 비교 대상 2~3개
- 행 단위: 가격 / 핏 / 소재 / 계절감 / 배송 / 반품 / 리뷰 요약
- 가장 유리한 지점 강조

#### Evidence / Reason Card
필수 정보:
- 반영된 조건
- 주요 근거 리뷰
- 핏/사이즈 주의사항
- 데이터 기준 시점

#### Feedback Card
필수 정보:
- 좋았던 이유 / 아쉬운 이유 선택 칩
- 예: 더 기본적으로, 더 저렴하게, 더 슬림하게, 이 조합은 별로

### 15.7 AI Progress / Working State

#### 목적
사용자가 “지금 AI가 무엇을 하고 있는지” 알게 해 불안을 줄인다.

#### UI 요소
- 현재 단계 이름
- 진행 바 또는 step list
- 반영 중인 조건 리스트
- 제외 조건 표시
- 짧은 상태 설명

#### 예시 단계
- 조건 정리 중
- 코디 후보 찾는 중
- 리뷰 요약 중
- 비교 포인트 정리 중

#### 금지
- "AI가 생각 중입니다..." 같은 모호한 문구만 단독 사용
- 긴 시간 아무 변화 없는 스피너

## 16. 픽핏 전용 UX 패턴

### 16.1 Onboarding Pattern
목표: 긴 설문이 아니라 빠른 결정 준비.

필수 질문 범주:
- 상황
- 예산
- 무드
- 선호 핏
- 체형 고민
- 피하고 싶은 요소
- 자주 입는 색

UI 원칙:
- 한 화면에 너무 많은 질문을 몰지 않는다.
- 1~2문항씩 점진적으로 받는다.
- 현재까지 반영된 조건을 요약 노출한다.

### 16.2 Recommendation Pattern
추천 결과는 카드 3개를 기본 단위로 한다.

구성:
- 코디 A / B / C
- 각각의 이유
- 가격 범위
- 리스크 요약
- 비교 진입

### 16.3 Comparison Pattern
비교는 보기 좋게 만들기보다 **빨리 선택하게 만드는 구조**여야 한다.

규칙:
- 2~3개 비교까지만 우선 지원
- 각 행은 의사결정 요소 중심
- 색상보다 레이블, 굵기, 아이콘으로 차이 강조

### 16.4 Reason / Evidence Pattern
추천 설명은 “감성 문장”이 아니라 구조화된 근거 중심.

권장 섹션:
- 이 조건을 반영했어요
- 이런 리뷰가 많았어요
- 이 점은 주의하세요
- 이런 상황에 특히 잘 맞아요

### 16.5 Purchase Handoff Pattern
MVP에서는 완전 자동 결제가 아니라 링크 이동이 기본이다.

따라서 UI는 아래를 명확히 해야 한다.
- 외부몰 이동 전 확인
- 옵션 상태
- 가격 / 배송 / 반품 핵심 요약
- 현재 시점 기준 정보인지 여부

## 17. 카피 / UX Writing 시스템

### 17.1 톤앤매너
- 짧고 명확하게
- 과장 없이
- 친절하지만 과하게 친근하지 않게
- 설명은 구체적으로
- AI를 만능처럼 말하지 않기

### 17.2 좋은 문장 예시
- 이 코디는 출근 상황과 슬림한 실루엣 선호를 반영했어요.
- 리뷰에서 정사이즈라는 의견이 많지만, 어깨가 넓으면 한 치수 업이 안전해요.
- 배송은 빠르지만 반품비가 높은 편이에요.

### 17.3 피해야 할 문장
- 당신만을 위한 완벽한 추천이에요.
- AI가 가장 좋은 답을 찾았어요.
- 무조건 잘 어울릴 거예요.

### 17.4 버튼 카피 원칙
- 추상 동사보다 구체 행동 사용
- 예: `추천 보기`, `비교하기`, `구매 링크 열기`, `조건 수정하기`

## 18. 프론트엔드 구현 가이드

### 18.1 토큰 우선 구현
컴포넌트 내부에 하드코딩된 색, radius, spacing을 넣지 않는다.
항상 디자인 토큰을 참조한다.

### 18.2 상태 클래스 분리
버튼, 칩, 카드의 상태는 최소 아래 상태를 분리한다.
- default
- hover
- pressed
- focus
- disabled
- loading

### 18.3 반응형 원칙
- 모바일 구조를 먼저 만들고 데스크탑 확장
- 데스크탑에서만 정보량을 늘리고 interaction logic은 유지

### 18.4 시맨틱 마크업
- 버튼은 button
- 네비게이션은 nav
- 비교표는 table 가능 여부 검토
- 리스트는 ul/ol 적극 사용
- 상태 설명은 aria-live 필요 시 적용

## 19. QA 체크리스트

### 19.1 브랜딩
- 로고 비율이 유지되는가
- 라임 포인트가 과하게 쓰이지 않았는가
- Brand Blue와 CTA Blue 역할이 섞이지 않았는가

### 19.2 타이포
- 본문 가독성이 확보되는가
- 제목과 본문 위계가 분명한가
- 숫자/가격이 빠르게 읽히는가

### 19.3 레이아웃
- 카드 내부 padding이 일관적인가
- 섹션 간 여백이 불규칙하지 않은가
- 모바일에서 정보가 과밀하지 않은가

### 19.4 컴포넌트
- 버튼 높이와 상태가 일관적인가
- 칩 선택 상태가 명확한가
- 인풋 에러 상태가 충분히 읽히는가

### 19.5 접근성
- 대비가 충분한가
- 인터랙션 hit area가 충분한가
- focus state가 보이는가
- 상태를 색만으로 전달하지 않는가

### 19.6 제품 적합성
- 화면이 “많이 보여주는 곳”처럼 보이지 않는가
- 추천 이유가 구조적으로 드러나는가
- 비교가 장식이 아니라 의사결정 도구로 작동하는가

## 22. 최종 요약

### 픽핏 디자인 시스템 한 줄 정의
**픽핏의 디자인 시스템은 패션 감도를 잃지 않으면서도, 구조화된 정보와 설명 가능한 추천으로 사용자의 구매 결정을 빠르게 끝내게 만드는 모바일 퍼스트 신뢰형 UI 시스템이다.**

### 꼭 기억할 것
- 패션앱처럼 보여야 하지만 패션 콘텐츠앱처럼 행동하면 안 된다.
- AI앱처럼 보여야 하지만 AI 데모처럼 과장되면 안 된다.
- 예쁜 추천보다 **빠른 결정**이 우선이다.
- 모든 컴포넌트는 “이게 사용자의 결정을 얼마나 쉽게 만드나”로 평가한다.

## 23. 참고 기준

이 문서는 다음 기준을 참고해 정리했다.

1. 프로젝트 내부 SSOT: `project.md`
2. 픽핏 확정 로고/컬러 시스템 이미지
3. W3C WCAG 2.2 접근성 기준
4. Material Design 3 접근성 및 인터랙션 밀도 가이드
5. Apple Human Interface Guidelines의 hit target 및 인터랙션 안정성 원칙

참고 메모:
- WCAG 2.2는 일반 텍스트 대비 4.5:1, 대형 텍스트 3:1을 기준으로 한다.
- WCAG 2.2는 24x24 CSS px 최소 타깃을 제시하지만,
  픽핏은 모바일 사용성과 오조작 방지를 위해 48x48px 수준을 제품 기본 기준으로 채택한다.
- Apple은 44x44pt hit region, Material은 48x48dp touch target을 권장한다.
